# 공통 원칙 (코딩 + 리뷰)

## SOLID + LoD
- SRP: 이 클래스/함수가 변경되어야 하는 이유가 둘 이상인가?
- OCP: 기존 코드 수정 없이 확장 가능한가?
- LSP: 부모 타입 자리에 넣어도 동작이 보장되는가?
- ISP: 사용하지 않는 메서드를 구현하도록 강제하는가?
- DIP: 구체 클래스에 직접 의존하는가?
- LoD: 체이닝(a.b.c.d)이 2단계를 초과하는가?
- 단, 클래스가 3개 이하 메서드면 간결함 우선. 과도 분리 금지.

## 구현 규칙
- 기존 코드 패턴을 따른다. 새 패턴 도입 시 근거 명시.
- 구현체가 하나뿐인 인터페이스를 만들지 않는다.
- 에러는 삼키지 않는다. catch 시 구체적 타입 + 의미 있는 처리.
- any 타입 금지. unknown + 타입 가드 사용.
- 구현 과정에서 발견한 legacy 코드(미사용 import, dead code, 폐기된 패턴)는 제거한다. 별도 지시가 없는 한 정리 방향으로. dead code 확인 시 diff만 보지 않고 프로젝트 전체를 탐색한다 — test, plugin, config 등 수정 빈도가 낮은 파일에서 참조하고 있을 수 있다.
