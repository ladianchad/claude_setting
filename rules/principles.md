# 공통 원칙 (코딩 + 리뷰)

## SOLID + LoD
- SRP: 이 클래스/함수가 변경되어야 하는 이유가 둘 이상인가?
- OCP: 기존 코드 수정 없이 확장 가능한가?
- LSP: 부모 타입 자리에 넣어도 동작이 보장되는가?
- ISP: 사용하지 않는 메서드를 구현하도록 강제하는가?
- DIP: 구체 클래스에 직접 의존하는가?
- LoD: 체이닝(a.b.c.d)이 2단계를 초과하는가?
- 단, 클래스가 3개 이하 메서드면 간결함 우선. 과도 분리 금지.

## 구현 규칙
- 기존 코드 패턴을 따른다. 새 패턴 도입 시 근거 명시.
- 구현체가 하나뿐인 인터페이스를 만들지 않는다.
- 에러는 삼키지 않는다. catch 시 구체적 타입 + 의미 있는 처리.
- any 타입 금지. unknown + 타입 가드 사용.

## 테스트 커버리지

모든 변경은 아래 두 기준을 **동시에** 충족해야 한다:

1. **Diff coverage ≥ 90%**: 이번 변경에서 추가/수정한 라인의 90% 이상이 테스트로 실행된다.
2. **변경된 파일 전체 coverage ≥ 90%**: 변경을 포함하는 각 파일의 라인 커버리지가 90% 이상이다.

기존 파일이 90% 미만이었다면 해당 파일을 90%로 끌어올리는 것은 현재 작업 범위에 포함된다 ("Pre-existing 문제 처리" 원칙과 일관). 사용자가 명시적으로 "커버리지 끌어올리기 제외"를 지시한 경우에만 면제된다.

### 예외 사유 (화이트리스트)
아래 사유에 한해서만 90% 기준에서 제외된다. 제외 적용 시 완료 보고에 **사유 + 해당 파일/심볼 경로**를 반드시 명시한다.

- **(a) Thin wrapper**: main entry, bootstrap, DI 컨테이너 설정, 순수 re-export 파일 등 테스트 가치가 없는 얇은 진입부.
- **(c) 자동 생성 코드**: protobuf, openapi-generator, prisma client, graphql codegen 등의 산출물. 생성 소스는 테스트 대상이 아니다.
- **(f) 사용자 명시 면제**: 사용자가 "커버리지 낮춰도 된다"고 명시적으로 지시한 경우.

위 3가지 외의 사유(복잡도, 시간 부족, 테스트하기 어려움, 프라이빗 헬퍼 등)는 예외 통로가 아니다. 테스트를 추가하여 기준을 충족시킨다.

### 측정 도구
프로젝트에 이미 설정된 커버리지 도구를 1차로 사용한다 (`package.json`, `pyproject.toml`, `Cargo.toml`, `Makefile`, `.pre-commit-config.yaml` 등에서 탐색). 프로젝트에 전혀 설정이 없으면 언어 표준 도구를 도입하고 도입 사실을 사용자에게 고지한다.

## Pre-existing 문제 처리 (기본: 포함)
작업 중 발견한 **pre-existing error, legacy 코드, dead code**는 사용자가 "건드리지 말라", "범위 밖", "무시" 등을 **명시적으로 지시하지 않은 이상 수정/정리 대상에 포함**한다. "지금 작업 범위가 아니다"를 자의적으로 판단해 방치하지 않는다.

대상 범위:
- **pre-existing error**: 빌드 경고, 타입 에러, 린터 경고, 런타임 에러, 테스트 실패, 잘못된 assertion 등 원래 있던 결함.
- **legacy 코드**: 폐기된 패턴, 미사용 import, 주석 처리된 코드, deprecated API 사용.
- **dead code**: 호출되지 않는 함수/클래스, 도달 불가능한 분기, 쓰이지 않는 변수/파라미터.

탐색 원칙:
- dead code 판정 시 diff만 보지 않고 프로젝트 전체를 탐색한다 — test, plugin, config, script, CI yaml 등 수정 빈도가 낮은 파일에서 참조하고 있을 수 있다. 확신이 없으면 제거 대신 TODO로 표시하고 보고한다.

처리 절차 (발견 시):
1. **현재 작업과 강결합 + 소규모** (수 줄 이내, 현재 변경과 같은 파일/함수): 같은 커밋에서 바로 수정.
2. **독립적이거나 중간 규모** (별도 파일/모듈, 수십 줄): **sub-task**로 처리. 별도 커밋으로 분리하되 같은 sprint/coding 세션에서 완료한다. 커밋 메시지 prefix는 `chore:` 또는 `fix:` 로 현재 기능 커밋과 구분한다.
3. **대규모** (구조 변경, 다중 파일 연쇄, 설계 검토 필요): 현재 작업을 오염시키지 않도록 **별도 `/sprint` 호출을 sequential 하게 실행**한다. 요구사항 파일을 즉석에서 작성하여 인자로 전달. 현재 작업은 발견 사항만 기록하고 진행한다.
4. 사용자가 명시적으로 "건드리지 말라"고 한 경우에만 위 절차를 건너뛰고 보고에만 포함한다.

모든 경우에 완료 보고에 **"발견한 pre-existing 이슈 + 처리 방식 (포함/sub-task/sequential/보류)"**을 명시한다.
