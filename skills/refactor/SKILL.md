# Refactor

프로젝트를 분석하여 리팩토링 대상을 식별하고, 설계 → 구현 → 검증까지 수행.
`/refactor`로 호출. 선택적으로 스코프(모듈명, 디렉토리, 파일경로)를 인자로 전달.

## 입력
- **스코프 미지정**: 프로젝트 전체를 분석하여 리팩토링 대상을 자동 식별.
- **스코프 지정**: 해당 범위만 집중 분석. 파일/디렉토리/모듈명 모두 가능.
- 사용자가 구체적 리팩토링 목표를 명시한 경우 (e.g. "이 클래스 SRP 위반 해소") Phase 1을 간소화.

## 스케일링 판단
Phase 1 완료 후 식별된 리팩토링 항목으로 규모를 판단한다:
- **소규모** (단일 파일, 10줄 미만): Phase 2 생략 → 바로 Phase 3.
- **중규모** (복수 파일 또는 10줄 이상, 인터페이스 변경 없음): 간소 설계 → 구현.
- **대규모** (인터페이스 변경, 모듈 분리/통합, 다중 파일 연쇄 변경): 전체 설계 → 구현.

---

## Phase 1: 프로젝트 분석 및 리팩토링 대상 식별

### 1-1. 구조 파악
1. 디렉토리 트리 스캔 + 진입점 식별.
2. 레이어 구조 파악: presentation → application → domain → infrastructure.
3. 의존 관계 추적 (import/DI 흐름). 순환 의존성 탐지.
4. 빌드/테스트 명령 확인 (package.json, Makefile, build.gradle 등).
5. 테스트 커버리지 현황 파악 (테스트 파일 존재 여부, 구조).

### 1-2. 코드 품질 분석
~/.claude/rules/principles.md 기준으로 체계적으로 점검한다.

**SOLID 위반 탐지**:
- SRP: 변경 이유가 둘 이상인 클래스/모듈 식별. 파일 크기, 메서드 수, import 다양성으로 1차 필터링 → 내용 확인.
- OCP: 조건 분기(if/switch)로 확장하는 패턴 식별.
- LSP: 상속 구조에서 부모 계약 위반 탐색.
- ISP: 인터페이스 구현체가 빈 메서드/throw로 구현하는 패턴.
- DIP: 구체 클래스에 직접 의존하는 상위 레이어 식별.
- LoD: 3단계 이상 체이닝 탐색.

**구현 규칙 위반 탐지**:
- 구현체가 하나뿐인 인터페이스.
- 에러를 삼키는 catch 블록 (빈 catch, console.log만 있는 catch).
- any 타입 사용.
- Dead code: 미사용 import, 호출되지 않는 함수/클래스. test, plugin, config 등 전체 탐색.
- 중복 코드: 유사한 로직이 복수 위치에 존재.

**구조적 문제 탐지**:
- 순환 의존성.
- God class / God module (과도한 책임 집중).
- Feature envy (다른 모듈의 데이터를 과도하게 참조).
- 레이어 위반 (하위 레이어가 상위 레이어를 import).
- 일관성 파괴 (동일 계층의 모듈이 서로 다른 패턴 사용).

### 1-3. 리팩토링 후보 목록 작성
각 후보에 대해:
- **위치**: 파일/클래스/함수.
- **문제 유형**: 위반 원칙 또는 코드 스멜 이름.
- **심각도**: Critical / Major / Minor.
  - Critical: 버그 유발 가능, 순환 의존, 레이어 위반.
  - Major: SOLID 위반, God class, 중복 코드.
  - Minor: 네이밍, dead code, 미사용 인터페이스.
- **영향 범위**: 이 코드를 참조하는 파일 수.
- **리팩토링 방향**: 1줄 요약 (e.g. "Extract Class로 책임 분리").

### 1-4. 우선순위 결정
- Critical → Major → Minor 순으로 정렬.
- 동일 심각도 내에서는 영향 범위가 넓은 것 우선.
- Critical + Major 항목을 리팩토링 대상으로 확정한다. Minor는 대상 항목 수정 시 자연스럽게 해소되는 경우만 포함.
- 사용자가 스코프를 인자로 제한한 경우 해당 범위 내 항목만 대상.
- 대상이 자명하면 바로 Phase 2로 진행한다. 영향 범위가 넓거나 판단이 갈리는 항목이 있으면 사용자에게 핵심 결정 사항을 제시하고 승인을 받는다.
- 확정 항목으로 스케일링 판단 → Phase 2 또는 Phase 3.

## Phase 2: 리팩토링 설계

→ ~/.claude/rules/round-agent-protocol.md 적용 (중규모/대규모).

### 소규모: 생략
Phase 1 분석 결과만으로 바로 Phase 3.

### 중규모: 간소 설계
- RefactorDesignRoundAgent subagent 1개를 생성하여 위임한다.
- Round Agent 내부: 단독 리팩토링 설계안 작성 + sub-subagent 1개로 검증.
- 검증 관점:
  - 리팩토링 후 기존 테스트가 깨지지 않는가.
  - 외부 인터페이스(public API)가 변경되는가. 변경되면 호출부 영향도 명시.
  - 리팩토링이 새로운 SOLID 위반을 만들지 않는가.
- PASS → 사용자 승인 후 Phase 3. FAIL → 수정 후 재검증 1회.
- 재검증도 FAIL → 사용자에게 제시하고 판단 요청.

### 대규모: 전체 설계
- Skill tool로 `/design`을 직접 호출하여 전체 설계 프로세스를 실행한다.
- `/design`이 내부에서 DesignRoundAgent를 활용하여 독립 설계 + 합의 + 검증을 수행한다.
- 설계안 제출 결과(합의안 + decision_rationale + impact_analysis)를 수신한다.

### 설계안 필수 포함 사항 (중규모/대규모 공통)
- **변경 명세**: 파일/클래스/함수 단위. 각 변경의 리팩토링 기법 명시 (Extract Class, Move Method, Replace Conditional with Polymorphism 등).
- **구현 순서**: 의존성 순. 특히 리팩토링은 중간 상태에서도 빌드/테스트가 통과해야 하므로, 각 단계가 독립적으로 검증 가능해야 한다.
- **안전망 전략**: 기존 테스트로 커버되지 않는 변경 부분에 대해 선행 테스트 작성 계획.
- **롤백 전략** (대규모만): 단계별 롤백 가능성.
- 설계안이 자명하면 Phase 3으로 자동 전환한다.
- 트레이드오프가 있거나 리팩토링 방향이 갈리는 경우에만 사용자에게 핵심 결정 사항을 제시하고 승인을 받는다.

### Phase 2 → Phase 3 전환 시 정보 전달
- **전달**: 확정 설계안 (변경 명세 + 구현 순서 + 안전망 전략) + 핵심 설계 결정 근거.
- **차단**: 대안으로 검토했다가 버린 설계안, 합의 과정.

## Phase 3: 리팩토링 구현

→ ~/.claude/rules/round-agent-protocol.md 적용 (중규모/대규모).

모든 변경은 ~/.claude/rules/principles.md를 준수.

### 핵심 원칙: 안전한 리팩토링
1. **Red-Green 확인**: 변경 전 기존 테스트가 모두 통과하는지 먼저 확인. 실패하는 테스트가 있으면 리팩토링 전에 사용자에게 보고.
2. **선행 테스트**: 설계안의 안전망 전략에 따라, 리팩토링 대상 코드의 현재 동작을 보호하는 테스트가 부족하면 먼저 작성.
3. **단계별 검증**: 설계안의 구현 순서 각 단계 완료 후 빌드/테스트 통과 확인. 중간 단계에서 실패하면 해당 단계를 수정.
4. **동작 보존**: 리팩토링은 외부에서 관찰 가능한 동작을 변경하지 않는다. 의도적 동작 변경이 필요하면 사용자에게 명시적 승인.

### 소규모 (10줄 미만)
바로 구현 → 빌드/테스트 통과 확인 후 완료.

### 중규모/대규모 (10줄 이상): `/coding` 호출

Skill tool로 `/coding`을 호출하여 구현을 위임한다.
- 전달: 확정 설계안 (변경 명세 + 구현 순서 + 안전망 전략) + 빌드/테스트 명령 + 아래 리팩토링 전용 검증 항목.
- `/coding`이 내부에서 병렬 구현 + 통합 + 병렬 리뷰 검증 루프를 수행한다.
- **리팩토링 전용 추가 검증 항목** (구현 요구사항에 포함하여 전달):
  - 리팩토링 전후 외부 인터페이스 동일성 확인 (public API 시그니처 비교).
  - 원래 식별한 코드 스멜/위반이 실제로 해소되었는지 확인.
  - 리팩토링으로 새로운 위반이 생기지 않았는지 확인.
- `/coding` 완료 후 결과(modified_files, summary)를 수신하여 Phase 4로.

## Phase 4: 완료 보고

### 4-1. 결과 매핑
- Phase 1에서 식별한 리팩토링 대상 목록 대비 결과:
  - 해소됨: 어떤 기법으로 해소했는지 1줄.
  - 미해소: 사유 (스코프 외, 사용자가 제외, 기술적 제약).

### 4-2. 변경 요약
- 변경된 파일 목록 + 핵심 변경 내용.
- 설계안 대비 달라진 점이 있으면 명시 및 사유.
- 추가/삭제된 테스트.
- 삭제된 dead code 목록.

### 4-3. 품질 개선 요약
- 해소된 SOLID 위반 수.
- 제거된 중복 코드 규모.
- 해소된 순환 의존성.
- 정리된 dead code 규모.

### 4-4. 잔여 사항
- 이번 리팩토링으로 해결하지 못한 항목.
- 후속 리팩토링 권고 사항 (있는 경우).

### 4-5. 커밋
- 커밋 여부를 사용자에게 확인. 자동 커밋하지 않는다.
- 커밋 시 리팩토링 단계별로 원자적 커밋을 권고 (사용자 선택).

## 원칙
- 모든 변경: ~/.claude/rules/principles.md 준수.
- 프로젝트에 `.claude/rules/workflow.md`가 있으면 해당 Git 워크플로우를 따른다.
- 기존 아키텍처를 존중. 리팩토링은 아키텍처를 개선하되, 근본적으로 다른 아키텍처로 전환하지 않는다 (사용자가 명시적으로 요청한 경우 제외).
- 과도 리팩토링 금지. 식별된 문제를 해소하는 데 필요한 만큼만.
- 동작 보존 원칙: 리팩토링은 외부 동작을 변경하지 않는다.
- Phase 간 전환 시 확정 산출물만 전달한다. 과정은 전달하지 않는다.
