# Technical Search

프로젝트에 맞는 라이브러리, 방법론, 패턴을 인터넷에서 탐색하고 평가하는 스킬.
`/search <질문 또는 주제>`로 호출.

## 입력
- 사용자의 질문 또는 탐색 주제를 인자로 받는다.
- 예: `/search 이 프로젝트에 맞는 state management 라이브러리`, `/search 대량 이미지 처리 파이프라인 방법론`

---

## Phase 1: 프로젝트 컨텍스트 분석

### 1-1. 기술 스택 파악
1. 디렉토리 트리 스캔 + 빌드 설정 파일 확인 (package.json, pyproject.toml, build.gradle, Cargo.toml 등).
2. 주요 의존성 목록 추출 + 버전 확인.
3. 언어, 프레임워크, 런타임 환경 식별.
4. 기존 아키텍처 패턴 파악 (레이어 구조, DI 방식, 상태 관리 방식 등).

### 1-2. 제약 조건 정리
- 언어/프레임워크 호환성 (e.g. React 18+, Python 3.11+, Node 20+).
- 기존 의존성과의 충돌 가능성.
- 라이선스 제약 (프로젝트 라이선스와 호환되는지).
- 번들 사이즈, 성능 요구사항 (해당되는 경우).
- 사용자가 명시한 추가 제약.

### 1-3. 탐색 방향 수립
- 사용자 질문을 프로젝트 컨텍스트와 결합하여 구체적 탐색 쿼리를 수립한다.
- 탐색 각도를 2~3개로 분기한다. 예:
  - 각도 1: 해당 프레임워크 생태계 내 솔루션
  - 각도 2: 프레임워크 무관 범용 솔루션
  - 각도 3: 최신 트렌드/신규 접근법

---

## Phase 2: 병렬 탐색

복수의 탐색 subagent를 병렬 생성한다. 각 agent는 서로의 존재를 알지 못한다.

### 각 탐색 agent에게 전달
- 프로젝트 컨텍스트 (기술 스택 + 제약 조건)
- 담당 탐색 각도
- 사용자 원본 질문

### 각 탐색 agent가 수행
1. **WebSearch**로 관련 라이브러리, 방법론, 패턴을 검색한다.
   - 검색 쿼리를 다양하게 변형하여 최소 3회 이상 검색.
   - 최신성을 위해 최근 1~2년 결과를 우선.
2. **WebFetch**로 유망한 결과의 공식 문서, GitHub 저장소, 블로그를 읽는다.
3. 각 후보에 대해 수집:
   - 이름 + 한줄 설명.
   - GitHub stars, 최근 커밋 날짜, 메인테이너 활성도.
   - 라이선스.
   - 프로젝트 기술 스택과의 호환성.
   - 주요 장점/단점.
   - 실사용 사례 또는 벤치마크 (있는 경우).

---

## Phase 3: 평가 및 통합

메인 agent가 모든 탐색 결과를 통합한다.

### 3-1. 중복 제거
동일 라이브러리/방법론이 여러 agent에서 발견된 경우 병합.

### 3-2. 적합성 평가
각 후보를 아래 기준으로 평가한다:

| 기준 | 가중치 | 설명 |
|------|--------|------|
| **프로젝트 호환성** | 높음 | 기술 스택, 의존성 충돌, 라이선스 |
| **성숙도** | 높음 | 버전 1.0+, 안정적 API, 프로덕션 사용 사례 |
| **유지보수 상태** | 높음 | 최근 6개월 내 커밋, 이슈 응답 속도, 메인테이너 수 |
| **문서 품질** | 중간 | 공식 문서, 예제, 마이그레이션 가이드 |
| **커뮤니티** | 중간 | GitHub stars, npm/PyPI 다운로드 수, Stack Overflow 답변 |
| **성능** | 조건부 | 벤치마크 결과 (해당되는 경우만) |

### 3-3. 순위 결정
- 적합성 상위 3~5개를 최종 후보로 선정.
- 각 후보에 **추천도** 부여: Strong / Moderate / Conditional.
  - Strong: 프로젝트에 바로 도입 가능, 리스크 낮음.
  - Moderate: 도입 가능하나 일부 트레이드오프 존재.
  - Conditional: 특정 조건 충족 시에만 적합.

## Phase 4: 보고

### 4-1. 결과 보고
각 후보에 대해:
- 이름 + 설명.
- 추천도 (Strong / Moderate / Conditional) + 근거.
- 프로젝트 도입 시 예상 영향: 변경 범위, 마이그레이션 비용.
- 공식 문서/저장소 링크.

### 4-2. 비교 요약
- 후보 간 핵심 차이점 비교표.
- 프로젝트 상황에서의 최종 권고 (1~2줄).

### 4-3. Design 연계
- 탐색 결과를 상태 파일로 저장한다: `{project_root}/.claude/state/search-{YYYYMMDD-HHmmss}.json`
- 스키마:
```json
{
  "skill": "search",
  "created_at": "...",
  "query": "사용자 원본 질문",
  "project_context": {
    "tech_stack": ["..."],
    "constraints": ["..."]
  },
  "candidates": [
    {
      "name": "...",
      "description": "...",
      "recommendation": "Strong | Moderate | Conditional",
      "rationale": "...",
      "links": ["..."]
    }
  ],
  "summary": "최종 권고 요약"
}
```
- **상태 파일 경로를 사용자에게 명시적으로 보고한다.**
- 다른 skill과의 연계는 **사용자가 명시적으로 상태 파일 경로를 전달할 때만** 이루어진다. 자동 탐색하지 않는다.
- 예: `/design 이 검색 결과 바탕으로 설계해 {search_state_file_path}`

---

## 원칙
- 존재하지 않는 라이브러리를 날조하지 않는다. WebSearch/WebFetch로 확인된 것만 보고.
- 검색 결과가 부족하면 "해당 영역에 적합한 후보를 찾지 못했다"고 솔직히 보고.
- 사용자 질문 범위를 벗어나는 불필요한 탐색을 하지 않는다.
- deprecated 라이브러리는 명시적으로 표기하고 대안을 제시한다.
